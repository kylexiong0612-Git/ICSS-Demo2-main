# Proposal: 可配置坐席工作流与工作台编排

**变更编号**: feat-configurable-agent-workflow-workbench
**日期**: 2026-04-15

## 问题陈述

当前系统的坐席处理链路和坐席工作台均为固定实现：

- 坐席链路仅支持固定的 `L1 / L2` 两级模型
- 任务模型使用 `level: 1 | 2` 表示处理层级，无法表达不同业务类型下的多级链路
- 坐席工作台页面为固定三栏结构，缺少按业务场景、链路阶段、角色进行界面编排的能力

这导致两个明显限制：

1. 无法为不同任务类型配置不同的坐席处理链路，例如“运营链路”为一线/二线，“系统报障链路”为一线/二线/三线
2. 无法为不同链路阶段提供差异化工作台，页面元素不能灵活组合，难以支撑运营、报障、复杂升级等场景
3. 客户侧“转人工”仅能创建通用任务，缺少对转人工意图的识别与分流，无法把任务准确送入对应类型的坐席链路
4. 顶部演示导航未直接体现“运营链路 / 报障链路”的阶段职责，且缺少报障 `L2 / L3` 的直达入口，不利于阶段化演示与验证

## 目标

- 支持定义多个坐席工作流模板，并为每个模板配置多个处理阶段
- 支持将任务分配到指定工作流，并在升级/流转时进入下一个配置阶段
- 支持在客户转人工时识别服务意图，并将任务自动路由到对应工作流的首阶段
- 支持配置化定义坐席工作台页面结构，实现页面区域、组件和参数的灵活组合
- 支持按“工作流 + 阶段”加载对应的坐席工作台配置
- 支持将顶部演示导航与工作流阶段命名保持一致，并补充报障链路 `L2 / L3` 的直接入口
- 保留当前双层坐席能力作为默认兼容方案，避免一次性破坏既有演示流程

## 非目标

- 本次不实现通用低代码引擎
- 本次不覆盖客户侧和管理大屏的全面可视化编排
- 本次不引入复杂审批流、并行节点、条件分支和 SLA 规则引擎
- 本次不处理完整 RBAC 权限体系，只在现有角色基础上按链路阶段进行最小控制

## 预期价值

- 让不同业务类型拥有独立的坐席处理链路，减少硬编码扩展成本
- 让转人工任务从“统一进入公共池”升级为“按意图精准分流到对应坐席链路”
- 让工作台按业务场景加载更合适的组件组合，提升坐席操作效率
- 让演示导航入口与工作流阶段定义保持一致，减少演示、培训和联调时的认知成本
- 为后续扩展更多链路类型、更多坐席层级、更多工作台组件打下统一配置基础
- 将“任务流转逻辑”和“页面呈现逻辑”从固定实现升级为配置驱动实现
